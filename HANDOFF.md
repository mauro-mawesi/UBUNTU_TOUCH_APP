## TL;DR

We're building a chat app for Ubuntu Touch that answers questions from a company's documents (RAG) and is meant to grow into a "company brain" — multiple knowledge topics + tool execution (SQL, CSV, Excel). It runs end-to-end today: chat works, persistence works, theme/i18n work, voice in (STT via self-hosted Whisper) and voice out (TTS via self-hosted Kokoro) work. Next direction agreed with the user: **multi-topic RAG first, then a function-calling tool harness**.

The app is a hybrid C++/QML clickable project. UT ships Qt 5.12 — that constraint shaped a lot of decisions.

## How we got here — chronological

### Stage 0: Empty repo
Started with a barebones Lomiri/QML "hello world" generated by `clickable create`. Pure QML builder (`pure-qml-cmake`). One `qml/Main.qml` with a "Hello World" label.

### Stage 1: Wire up the RAG flow (still pure QML)
The user had infrastructure already running on `172.28.18.200`:
- **ChromaDB** (port 8000) with two collections of company docs
- **Ollama** (port 11434) with `nomic-embed-text` (768-dim, matches Chroma)
- He has an **OpenRouter** API key and wants **Gemini 2.5 Pro** as the LLM

Built the pipeline in pure QML:
- `js/ChromaClient.js` — embed user query via Ollama, query Chroma for top-K
- `js/OpenRouterClient.js` — SSE streaming chat completions via XMLHttpRequest with `readyState=3` polling (the canonical way to do streaming in QML without going to C++)
- `js/RagOrchestrator.js` — glue: retrieve → build context block → stream
- `ChatPage.qml`, `SettingsPage.qml`, `MessageBubble.qml`, `SuggestionCard.qml`
- Persistent settings via `Qt.labs.settings` (URLs, API key, collection ID, model)

The collection `ssp-docs` (id `7e001a88-…`, 322 docs, client-side embeddings) became the default because we have `nomic-embed-text` locally and the dimensions match. The other collection (`ssp-docs__gemini__…`) has more docs but requires a Gemini Embeddings API key we don't have.

### Stage 2: Make it look like an app, not a demo
The user wanted a modern, configurable visual identity:
- **Theme system** in `AppTheme.qml`: dark/light + 6 accent palettes (indigo, cyan, emerald, violet, rose, sunset). All colors are derived properties of `{mode, presetIndex}`. Gradient backgrounds (3 stops) and gradient user bubbles. Settings UI lets the user switch live.
- Property injection from `Main.qml` → pages → components. Initially tried a `pragma Singleton`; it broke (see Gotcha #4 in CLAUDE.md). Property injection has been bulletproof.

### Stage 3: Multi-language (en / es / nl) with **live** switching
First attempt: standard gettext (`i18n.tr()` + `.po` files). Built and worked, but `LANGUAGE` env var has to be set before Qt starts — **no runtime switching** in Qt 5.12.

Pivoted to a custom system: `AppI18n.qml` is a `QtObject` with a `language` property and a `tr(key)` function that reads from `js/I18nData.js` (a JS dictionary). Bindings using `i18nApp.tr("X")` re-evaluate when `i18nApp.language` changes because QML's binding tracker observes property reads during binding evaluation.

Note: the type was originally called `I18n`. **Lomiri.Components 1.3 already exports a singleton named `I18n`** — naming collision crashed the app with the cryptic "Singleton object" error. Renamed to `AppI18n`.

The `po/` directory and `.mo` files still exist because the build chain wires them up. They're unused by the running app; they could be removed but aren't blocking anything.

### Stage 4: Persistence + sidebar
`js/Store.js` wraps `Qt.LocalStorage` (SQLite) with conversations + messages tables. `components/ConversationList.qml` is the sidebar — pinned in wide mode (≥ `gu(80)`), slide-in drawer in narrow mode. Toggle is the `navigation-menu` icon in the header, **always visible** (an earlier iteration only showed it in narrow mode; user couldn't hide the sidebar in wide mode and reported it).

Conversation flow: starts with no conversation (`currentConvId = -1`) → first user message creates a row with title derived from the first ~60 chars. Streaming messages are persisted up front (empty row) and updated on `onDone`. Persistence is wrapped in `try/catch` everywhere so a DB hiccup never blocks the LLM call.

### Stage 5: Voice — the big architectural shift
The user asked for STT + TTS. Audio capture in **pure QML on Qt 5.12 is essentially not viable** (`MediaPlayer` plays, doesn't record; `AudioRecorder` is not exposed as a QML element; the only audio-capture hack is via `Camera{}`, which is ugly).

Decision: **migrate the builder from `pure-qml-cmake` to `cmake`**. This was the highest-impact structural change in the project so far. Touched:
- `clickable.yaml` — `builder: cmake`, `kill: ragassistant`
- `ragassistant.desktop.in` — `Exec=ragassistant` (binary, no longer `qmlscene`)
- `ragassistant.apparmor` — added `audio` + `microphone` policy groups
- `CMakeLists.txt` — `add_subdirectory(src)`
- new `src/` with `CMakeLists.txt`, `main.cpp`, classes

**Critical detail in `main.cpp`**: `MainView` (Lomiri.Components) inherits from `Item`, not `Window`. `QQmlApplicationEngine` won't create a window from a non-Window root and the app starts invisibly. **Use `QQuickView` with `SizeRootObjectToView`** — that's what `qmlscene` did under the hood.

Server side:
- **Whisper** was already running on `:3011` (OpenAI-compatible) — discovered when I tried to deploy one and `docker run` failed with a name conflict. Confirmed it works for es/en/nl.
- **Kokoro** TTS deployed fresh on `:8880` (OpenAI-compatible).

C++ classes added:
- `AudioRecorder` (`QAudioRecorder`) — codecs/containers auto-detected at runtime because the default settings combo (`audio/PCM` + `wav`) wasn't accepted by the GStreamer plugins inside clickable's Docker. Preference order: WAV → OGG → MP3 → first available.
- `WhisperClient` — `QHttpMultiPart` upload to `/v1/audio/transcriptions`. Multipart from JS would have been painful.
- `TtsClient` — JSON POST to `/v1/audio/speech`, save MP3 to `CacheLocation`, emit `audioReady(filePath)`. QML's `MediaPlayer` plays the file.

UI additions:
- Mic button in input area, push-to-talk style. Pulsing red ring while recording, spinner while transcribing. Placeholder text changes to "Listening…" / "Transcribing…".
- Speaker button on assistant bubbles. Tap to play, tap again to stop.
- Auto-speak toggle in Settings.

### Stage 6: Polish iterations the user did
Between sessions / in parallel, the user added several niceties (visible in the file headers):
- **Splash screen** with `minVisibleMs` so it doesn't flash by on a fast cold start.
- **Phase tracking** during streaming — "Searching context…" while retrieving from Chroma, "Thinking…" between retrieval and first token. Drives a label next to the typing-wave indicator.
- **Copy button** on every bubble (F1) using a hidden `TextEdit` as clipboard.
- **Source chips** that expand into a snippet panel showing the actual chunk text (F7). Huge for trust.
- **Conversation search** in the sidebar.
- **Connectivity test panel** in Settings (per-service ping with status pill).
- **Wave animation** for the typing indicator (Translate transform + opacity, staggered).

```bash
clickable desktop                       # builds + runs in Docker; AUDIO DOES NOT WORK HERE
clickable install --arch arm64          # on a connected UT phone
clickable launch
```

Always clear QML cache before re-running if you touched singletons / pragmas:
```bash
rm -rf /home/mauricio/.clickable/home/.cache/ragassistant.ragassistant/qmlcache
```

### Server services (`172.28.18.200`)
| Service | Port | Endpoint |
|---|---|---|
| Chroma | 8000 | `/api/v2/tenants/default_tenant/databases/default_database/collections/.../query` |
| Ollama | 11434 | `/api/embeddings` (model `nomic-embed-text`) |
| Whisper | 3011 | `/v1/audio/transcriptions` (multipart) |
| Kokoro | 8880 | `/v1/audio/speech` (JSON, returns MP3) |

SSH access as `mauricio@172.28.18.200` with the user's default key.

### Where to look first when something breaks
- **Build fails** → look at the lines around `Built target ragassistant_autogen` and the last `[NN%]`. CMake errors usually point precisely at the file.
- **Window doesn't appear** → check `[main] loading QML from:` log. If logs from `[AudioRecorder] codecs available` appear but no window, it's a Qt issue in QML (usually a type that doesn't exist in 1.3 — e.g., `ThinDivider`).
- **Labels render as empty strings** → `i18nApp` not propagated to that component. Pass it as a property.
- **"Singleton object" error** → a QML type collides with a Lomiri-registered singleton. Rename it.
- **Mic/TTS don't work in desktop mode** → expected. Test on phone.

## Roadmap — where we want to go

The user's vision: this app becomes the **brain of a company** — multiple knowledge topics + ability to actually execute things (queries, files, web). Below is the agreed direction.

### Phase 1 — Multi-topic RAG  *(next session — not yet started)*
- New SQLite table: `topics(id, name, collection_id, color, icon, system_prompt_addon)`
- UI: topic chip in the chat header; tap to switch
- Each `conversations` row binds to a `topic_id` (we can repurpose the existing `collection_id` column)
- **Auto mode**: a small extra LLM call classifies each query and picks the right topic. Or: query all topics in parallel and re-rank — "RAG fusion".
- Settings: CRUD over topics

This phase gives the most immediate user-visible value with the lowest architectural risk. It doesn't touch the LLM pipeline shape.

### Phase 2a — Function-calling tool harness POC
- OpenRouter supports OpenAI-style `tools` parameter; Gemini 2.5 implements it natively
- Implement the loop:
  1. Send request with `tools` array
  2. LLM responds with `tool_calls` (instead of content)
  3. App executes the tool
  4. App appends `role: "tool"` message with the result
  5. Re-prompt the LLM
  6. LLM produces final content
- Start with **one dummy tool** (`get_current_time` or `calculate`) to validate the loop end-to-end
- **New UI**: collapsible "tool call" bubble showing the tool name, args, and result. Critical for user trust — if tools run invisibly, users won't believe the answers.

### Phase 2b — Real tools via FastAPI backend
A new service on `172.28.18.200` exposes tool endpoints:
- `/tools/sql` — configured DB connections, sandbox `SELECT`-only by default
- `/tools/csv` — DuckDB or pandas over user-provided CSVs
- `/tools/excel` — `openpyxl`, returns sheet/range JSON
- `/tools/web` — search + scrape (optional)

Frontend has a tool registry (`js/tools/registry.js`) — each entry: name, description, JSON Schema for parameters, an `execute(args)` function that either runs locally in JS or POSTs to the backend.

### Phase 3 — MCP migration  *(optional, longer term)*
Once there are 3+ tools, migrate to **Model Context Protocol** (Anthropic, open standard). One client, many independent tool servers. Scales better than maintaining a custom registry inside the app.

### Other directions the user expressed interest in (lower priority)
- Token / cost counter per turn (OpenRouter returns `usage`)
- Inline citations `[1]`, `[2]` that link to the source chips
- Export conversation to Markdown
- "Pin" of important messages
- Share conversation via UT content-hub

## Sharp edges that bit us — short list

If you're about to touch these areas, **read CLAUDE.md's Gotchas section first**. Quick callouts:

1. **`pragma Singleton` is broken with `import "." 1.0` in Qt 5.12.** We use property injection.
2. **`I18n` collides with Lomiri's built-in type.** Ours is `AppI18n`.
3. **`MainView` is not a `Window`.** Must use `QQuickView` in main.cpp.
4. **QML disk cache stays even after you delete pragmas.** `rm -rf` the cache before testing structural changes.
5. **`anchors.centerIn` on a `ColumnLayout` loops.** Use `horizontalCenter` + `verticalCenter` separately.
6. **`XDG_RUNTIME_DIR` unset in clickable's Docker.** Set it in `main.cpp` for PulseAudio.
7. **Audio devices not in clickable desktop's container.** Test on phone.
8. **`Store.*` calls are wrapped in try/catch.** Don't unwrap them — persistence is best-effort, chat flow must never block on the DB.
